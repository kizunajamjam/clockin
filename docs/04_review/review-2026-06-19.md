## 2. 機能レビュー

### F-1 🔴 日払い交通費の二重計上 — ✅修正済み
**事象**: マイグレーション `20260618000004_multi_punch_per_day.sql` で `attendances` の
`UNIQUE(shop_id, staff_id, date)` を撤廃し、1日に複数の出退勤レコードを許可した。
一方 `calcMonthlyPayroll`（[src/lib/payroll.ts](../../src/lib/payroll.ts)）は
レコード単位で `calcDailyPayroll` を呼び、`transport_fee_type='daily'` の交通費を
**レコードごと**に加算していた。そのため同日に2回打刻したスタッフは交通費が日数分でなく
打刻回数分支払われる（=過払い）。給与という金銭に直結する実害バグ。

**修正**: `calcMonthlyPayroll` で同一 `date` の2件目以降は交通費を0にし、
`total` から差し引くようにした。日払い交通費は「実出勤日数 × 単価」になる。

### F-2 🟡 出勤日数の過大カウント — ✅修正済み
**事象**: 同上の理由で `payroll.days.length`（owner側明細「出勤日数 N日」）が
打刻レコード数になり、同日複数打刻で水増しされていた。

**修正**: `MonthlyPayroll` に `work_days`（distinct date 件数）を追加し、
[payroll/[staffId]/page.tsx](../../src/app/(protected)/shops/[shopId]/payroll/[staffId]/page.tsx) の表示を差し替え。

### F-3 🟡 シフト画面の週境界TZズレ — ✅修正済み
**事象**: [shifts/page.tsx](../../src/app/(protected)/shops/[shopId]/shifts/page.tsx) で
`new Date()` の `getDay()/getDate()` を用いて週始めを算出していた。
Cloudflare Pages / Vercel など UTC サーバーでは、JST 早朝（00:00〜09:00）に前日扱いとなり、
表示週・取得範囲が1日ズレうる。`shift_requests` の範囲も `toLocaleDateString('sv-SE')`（TZ未指定）で
サーバーTZ依存だった。

**修正**: `Asia/Tokyo` で当日を確定し、曜日計算用のUTCアンカーで週始め（さらに日曜始まり店舗の
先頭日曜も取りこぼさないよう前日曜）を算出。`shifts` クエリは `+09:00` オフセット付き ISO で範囲指定、
`shift_requests` は `YYYY-MM-DD` 文字列で範囲指定するよう統一。

### F-4 🟢 React key 重複 — ✅修正済み
複数打刻で同一 `date` が複数行になると `key={day.date}` が重複し React 警告。
owner/staff 双方の給与明細で `key={`${date}-${index}`}` に変更。

### F-5 🟢 死にルート削除 — ✅削除済み
招待完了のリダイレクト先が `/punch/setup-complete` → `/punch-complete` に移行済みで、
旧ルートはどこからもリンクされず、ミドルウェアの公開パスからも外れている（=アクセスすると
ログインへ弾かれる）ため削除した。

### F-6 🟢 スタッフ側 payroll / shifts のTZ未固定 — 記録のみ
[punch/payroll/page.tsx](../../src/app/punch/payroll/page.tsx)・[punch/shifts/page.tsx](../../src/app/punch/shifts/page.tsx)
も `new Date()` 基準で範囲算出している。ただし「今月」「今日から28日」という前方向の窓であり、
TZズレの影響は端日のみで軽微。将来 `Asia/Tokyo` 固定に統一するのが望ましい。

### F-7 🟢 ダッシュボードの出勤人数 — 記録のみ
[dashboard/page.tsx](../../src/app/(protected)/dashboard/page.tsx) の「出勤中N名/退勤N名」は
`attendances` レコード単位の集計。複数打刻時、同一スタッフが「出勤中1 + 退勤済1」を持つと
両方にカウントされ実人数とズレうる。実害は表示のみ。`staff_id` で distinct すると正確になる。

## 3. セキュリティレビュー

### S-1 🔴 希望シフト提出のIDOR — ✅修正済み
**事象**: [punch/shifts/actions.ts](../../src/app/punch/shifts/actions.ts) の `upsertShiftRequest` は
クライアントから渡る `shop_id` を検証せず、`createAdminClient()`（service_role、RLS迂回）で
`shift_requests` に upsert していた。ログイン済みスタッフは、自分が所属しない任意店舗（別組織含む）の
`shop_id` を送って希望シフトを作成でき、その店舗オーナーの画面に身に覚えのない希望が現れる
（データ汚染・軽度の情報撹乱）。

**修正**: `shop_staff` で `(staff_id, shop_id, is_active=true)` の所属を必須検証するガードを追加。

### S-2 🔴 キオスクPINのブルートフォース耐性なし — ✅修正済み
**事象**: [kiosk/[shopId]/actions.ts](../../src/app/kiosk/[shopId]/actions.ts) の `punchTablet` は
4桁PIN（1万通り）を検証するが、**試行回数制限・ロックアウト・失敗遅延がなかった**。
キオスクページ [kiosk/[shopId]/page.tsx](../../src/app/kiosk/[shopId]/page.tsx) は未認証アクセス可で、
スタッフ氏名と `staff.id` を一覧表示するため、`shopId` を知る者は特定スタッフへ
オンライン総当たり（最大1万リクエスト）で他人になりすまし打刻が可能だった。

**修正**: `staff` に `pin_failed_count` / `pin_locked_until` を追加（マイグレーション
`20260619000000_pin_lockout.sql`）し、`punchTablet` に **staff単位のロックアウト**を実装。
連続 `MAX_PIN_ATTEMPTS=5` 回失敗で `PIN_LOCK_MINUTES=5` 分ロック、ロック中は検証せず拒否。
成功時・期限切れ時にカウンタをリセット。これにより総当たりは「5分あたり最大5回」に制限される
（PBKDF2 10,000回のコストと併せ実質的に総当たり不可）。CloudflareのEdge環境では
インメモリ状態を持てないため、状態はDB（staff行）で管理する。

**追加強化（2026-06-19 実装）**:
- **PIN桁数を可変（4〜6桁）**に。スタッフ作成/編集で任意の長さを設定でき、キオスクは
  4〜6桁を受け付ける（`/^\d{4,6}$/`）。桁数を増やすほど候補数が増え総当たり耐性が上がる。
- **打刻監査ログ＋IP単位スロットリング**（`punch_attempts` テーブル / migration
  `20260619000001`）。全打刻試行（成功/失敗・IP）を記録し、同一IPからの失敗が
  `IP_WINDOW_MINUTES=10` 分間に `IP_MAX_FAILS=20` 件でブロック（staff横断の総当たり対策。
  staff単位ロックアウトと二段構え）。Cloudflareの `cf-connecting-ip` 等からIP取得。外部サービス不要。

**残る改善余地（任意）**: キオスク端末トークンバインド・監査ログのオーナー閲覧UI。

### S-3 🟡 service_role 全面利用 — ✅認可ヘルパに集約
ほぼ全ての Server Action / Route Handler が `createAdminClient()` で RLS を迂回し、
**手動の認可チェックに全面依存**していた。認可チェックを1つ書き忘れるだけで全権限バイパスになる
（S-1 はまさにその実例）。

**修正**: 認可を共通モジュール [src/lib/auth.ts](../../src/lib/auth.ts) に集約した。
- `getAuthedStaff()` — ログインユーザー→staff 解決（+ `admin` クライアント同梱）
- `getAuthedStaffForShop(shopId)` — 上記 + `shop_staff` のアクティブ所属を検証
- `getOwnerShop(shopId)` — 組織所有者による店舗所有を検証

以下を全てヘルパ経由に統一し、各所での検証漏れを構造的に防止:
[punch/shifts/actions.ts](../../src/app/punch/shifts/actions.ts)・
[punch/actions.ts](../../src/app/punch/actions.ts)（punchSmartphone）・
[api/push/subscribe/route.ts](../../src/app/api/push/subscribe/route.ts)・
[owner shifts/actions.ts](../../src/app/(protected)/shops/[shopId]/shifts/actions.ts)
（重複していたローカル `getOwnerShop` を共通実装へ集約）。

**運用規約（継続）**: service_role を使う新規アクションは必ず `src/lib/auth.ts` のヘルパで
本人・所属・所有を検証すること。可能な経路はRLS有効クライアントへ寄せると二重防御になる。

### S-4 🟢 PIN関連の細部 — ✅修正済み
- `verifyPin`（[src/lib/pin.ts](../../src/lib/pin.ts)）を**定数時間比較**（XOR畳み込み）に変更。
- PBKDF2 反復回数を 10,000 → **100,000** に引き上げ。既存ハッシュ（10,000回）を壊さないよう
  保存形式を `v2$<iterations>$<hex>` のバージョン付きにし、旧形式（`$`なし）も検証可能。
  打刻成功時に `needsRehash()` が true なら現行パラメータへ**透過的に再ハッシュ**して昇格する。
- 補足: 4桁PINは候補1万通りのため、DB漏洩時の防御は反復回数よりS-2のロックアウト/将来の桁数増が本質。

  認証必須で、`user.id` から自分の `staff` を引いて購読を結びつけており、他人の購読を操作できない。
- ✅ `deleteShiftRequest` / `updateShiftRequestStatus` は staff_id / owner で適切にスコープ済み。
- 🟢 VAPID鍵未設定時 `sendPushNotification` はスキップ（[src/lib/push.ts](../../src/lib/push.ts)）。
  本番では `VAPID_*`・`NEXT_PUBLIC_VAPID_PUBLIC_KEY`・`VAPID_EMAIL` の設定が必要。

## 5. 全体監査（第2回・2026-06-19）

差分外の既存コードも含めてアプリ全体を通読した結果。**「service_role で取得・更新する際、所有者検証はしているが、書き込み先を `id` だけで指定し組織/店舗スコープを付けていない」**というIDOR系の見落としが複数見つかった（S-3の構造的リスクの実例）。

### R-4 🔴 最高: 店舗詳細ページに認可チェックなし — ✅修正済み
[shops/[shopId]/page.tsx](../../src/app/(protected)/shops/[shopId]/page.tsx) は所有者検証なしで
`shopId` から店舗・スタッフ一覧・**`invite_token`（招待URL）**を表示していた。`(protected)` で
認証は必須だが認可がないため、**任意のログインユーザー**（別組織オーナー/スタッフ）が
任意店舗の招待URLを取得でき、**未登録スタッフのアカウントを乗っ取れる**状態だった
（兄弟ページは全て所有者検証済みで、本ページのみ欠落）。
**修正**: `getUser()` + 組織所有者検証を追加し、非所有者は `notFound()`。

### R-1 🔴 高: 勤怠修正のテナント越境（IDOR） — ✅修正済み
[attendance/actions.ts](../../src/app/(protected)/shops/[shopId]/attendance/actions.ts) `correctAttendance` は
オーナー検証後、`attendances` を `.eq('id', attendanceId)` のみで更新していた。
自分が所有する `shopId` ＋ 他店舗の `attendanceId` を送れば**他組織の勤怠を改ざん**できた。
**修正**: 更新に `.eq('shop_id', shopId)` を追加（検証済み店舗に限定）。

### R-3 🔴 高: スタッフ更新のテナント越境（IDOR） — ✅修正済み
[staff/[staffId]/actions.ts](../../src/app/(protected)/shops/[shopId]/staff/[staffId]/actions.ts) `updateStaff` は
`staff` を `.eq('id', staffId)` のみで更新していた。他組織の `staffId` を送れば
氏名・性別・年収アラート・**PIN**を改ざん可能（他人のPIN変更→なりすまし打刻）。
**修正**: 更新に `.eq('organization_id', shop.organization_id)` を追加。
（`shop_staff` 側の更新は `shop_id`+`staff_id` でスコープ済みのため元から安全）

### R-2 🟡 中: プロプランのスタッフ上限 — ✅修正済み
[staff/new/actions.ts](../../src/app/(protected)/shops/[shopId]/staff/new/actions.ts) `createStaff` は
`TODO` のままプロ判定をせず常に5名上限。要件・課金画面では「プロ=無制限」と表記しているため不整合。
**修正**: `subscriptions.status` を確認し、プロ（active/trialing）は上限チェックをスキップ。

### R-5 🟡 中: フリープランの店舗数制限が未実装 — ✅修正済み
[shops/new/actions.ts](../../src/app/(protected)/shops/new/actions.ts) `createShop` は店舗数を制限していなかった。
課金画面の「フリー=1店舗 / プロ=無制限」表記に合わせ、プロでなければ既存店舗1件以上で作成を拒否。
判定は共通ヘルパ `isOrgPro`（[src/lib/plan.ts](../../src/lib/plan.ts)）を使用。

### F-8 🔴 高: 予定人件費の深夜割増判定が反転 — ✅修正済み
[ShiftCalendar.tsx](../../src/app/(protected)/shops/[shopId]/shifts/ShiftCalendar.tsx) `calcShiftCost` は
`night_rate_included` の真偽が [payroll.ts](../../src/lib/payroll.ts) と**逆**だった
（`true`=割増込みなのに割増を加算、`false`=割増対象なのに加算せず）。全スタッフで予定人件費が誤表示。
**修正**: payroll.ts と同じ規約（`true`は一律時給）に統一し、深夜帯（22:00〜5:00）を**1分刻み**で
判定（旧実装は15分近似＋比率補正だった）。確定給与計算と整合。

### R-2の共通化 / signup堅牢化（併せて対応）
- `isPro` 判定を [src/lib/plan.ts](../../src/lib/plan.ts)（`isProStatus` / `isOrgPro`）に集約し、
  dashboard・billing・owner payroll・staff/new・shops/new の重複インライン判定を置換。

### C-1 🟢 `year_month` の月部分を検証していない — ✅修正済み
**事象**: [payroll/[staffId]/actions.ts](../../src/app/(protected)/shops/[shopId]/payroll/[staffId]/actions.ts)
`upsertCustomRecords` の検証が `/^\d{4}-\d{2}$/` のみで、存在しない月（例: `2026-13`）の入力を拒否しなかった。
フロント側では `new Date()` 由来の値しか送らないため実害は限定的だが、Server Action は直接呼び出し可能なため
入力検証として不十分だった。

**修正**: 月部分（`yearMonth.slice(5,7)`）を数値化し、1〜12の範囲チェックを追加。
